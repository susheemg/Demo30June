Deterministic-firstEvery workflow produces a correct outcome with no live AI model. AI enriches; it never gates.The platform cannot be taken down by a provider outage; outputs are auditable.

Data flow

Three flows characterise the platform: the synchronous request lifecycle, the ProAssess assessment pipeline with its atomic auto-record chain, and the governed AI call path with its deterministic fallback.

3.1 Request lifecycle

Figure 3 — Synchronous request lifecycle

3.2 ProAssess assessment flow & atomic auto-record chain

When an assessor registers a completed ProAssess, the system executes a deterministic, atomic chain of 11 database writes — all succeed or all roll back. This is the platform’s self-maintaining data model in action: gaps become findings, the vendor is enrolled for monitoring, and the risk profile is refreshed without any manual step.

Methodology grounding (v4.5.0). Both the conversational assessor (BRO Chat) and the autonomous assessor (ProAssess) are guided by the firm’s methodology held in the Methodology folder — the active rows of the methodology_doc table, managed through the admin-only Methodology Library. At assessment time the platform calls methodology.methodology_directive(session), which concatenates every active methodology document (capped, prefixed “AUTHORITATIVE METHODOLOGY DOCUMENT — you MUST comply strictly”) and injects it into the agent system prompt via agents.build_system_prompt(…, methodology=…). BRO Chat (bro_app turn handler) and the AI-managed ProAssess path (master_service.run_proassess_autonomous → _proassess_ai) both resolve from this single source; the LangGraph 8-stage representation resolves the same directive via langgraph_flow._resolve_methodology(session). When the folder is empty, all paths fall back to the identical BEST_PRACTICE_DIRECTIVE (PRA SS2/21, EBA, DORA, FSB, NIST SP 800-161). The deterministic ProAssess scoring engine is rules-based and encodes the methodology weights directly, so it is unaffected by prompt grounding. Note: the firm’s methodology belongs in the Methodology Library, not the general Documents store, which holds per-vendor evidence and does not steer methodology.

Brata deploys as a single Render web service with push-to-deploy from GitHub. The runtime is deliberately minimal: one process, one database, zero required environment variables for a working instance. The architecture scales by configuration, not rewrite.

5.3 Health & readiness

GET /healthz returns platform version with no auth (Render health check). GET /readyz executes SELECT 1 to confirm DB connectivity before the load balancer routes traffic. Both are sub-2ms at current scale.

6.3 Audit trail & evidentiary record

Every state-changing action writes an immutable, hash-chained audit record: actor, UTC timestamp, action type, and a JSON payload of changed values. The audit log cannot be edited or deleted through any platform interface. Together with immutable assessment snapshots, the document store with full provenance, the AI ledger, and the finding lifecycle history, it constitutes the documented TPRM record required by DORA Article 28, FCA SYSC 8, and PRA SS2/21 — designed to answer, at any point in time: “what did you know about this vendor, and when?”

✓
Regulatory alignmentThe control set maps directly to the operational-resilience and outsourcing expectations of DORA, the PRA, the FCA, and UK GDPR Article 28 — the platform is built to produce regulatory evidence as a by-product of normal operation.

8.3 Multi-tenancy roadmap

The platform is designed for multi-tenancy as a phased, configuration-driven evolution. Phase 1 (the data layer) has been designed and prototyped: a tenancy.py resolution layer, tenant_id on all data tables, per-tenant ID sequences, Postgres RLS as defence-in-depth, and a tenant-resolution layer engineered so that database-per-tenant becomes a configuration change for Tier-1 regulated buyers who demand physical isolation. Subsequent phases add per-endpoint query scoping, per-tenant auth/SSO, per-tenant methodology and AI budgets, and isolation testing.

PhaseScopeEffort
P1 — Data layerPostgres + tenant_id on all tables + RLS + per-tenant IDs3–4 wks
P2 — ApplicationTenant context injection + query rewrite on 339 endpoints3–4 wks
P3 — Auth & identityPer-tenant users, scoped JWTs, SSO/OIDC2–3 wks
P4 — Shared servicesPer-tenant methodology, prompts, AI budget, webhooks1–2 wks
P5 — HardeningIsolation test suite, billing hooks, offboarding1–2 wks
ⓘ
Design intentItems 1–6 of the gap register are engineering hygiene that remove risk; item 7 is an architecture refactor; item 8 begins continuous model evaluation. None requires rework of the core architecture — tenancy, Postgres, SSO, and regulator-pack automation all slot into the existing design.

SOP-01

Introduction & scope

This Standard Operating Procedure governs all activities on the Brata TPRM platform. It is the authoritative reference for assessors, business leads, executives, auditors, and technology teams. It supersedes any prior process documentation for platform use.

Scope. All third-party relationships managed through the platform across all legal entities and jurisdictions. This covers initial onboarding, engagement assessment, contract review, ongoing monitoring, incident management, and exit planning.

Audit use. Every rule corresponds to code behaviour in v4.2.0. Function names, module paths, and line references are provided. Where a rule states the system "automatically" performs an action, that action executes without human intervention in the same database transaction and is logged to the immutable audit trail.

SOP-02

Core principles

P-01Deterministic-first, AI-augmented
Every workflow produces a deterministic outcome without a live AI model. AI enriches outputs; it never gates process steps. When AI is unavailable the system falls back to its rule engine and labels the output accordingly. The assessor's signed-off judgement is the system of record.
P-02Single source of truth per record
Each vendor, engagement, assessment, finding, contract and document has exactly one authoritative record identified by a system-generated ID: VEN-xxxxxx / ENG-xxxxxx / ASM-xxxxxx / FND-xxxxxx / CTR-xxxxxx / ART-xxxxxx. All inter-process references use these IDs.
P-03Risk-averse defaults
When the system cannot establish a material fact it resolves in the most risk-averse direction. Unscored criticality parameters are treated as 5 (maximum). An unknown country is treated as potentially sanctioned. An absent contract clause is treated as missing. An unverified control provides no residual risk reduction.
P-04Every action is audited
Every state-changing action is written to the immutable audit log: UTC timestamp, actor username, action type, and a JSON payload of changed values. The log cannot be edited or deleted through the platform interface.
P-05Third-party content isolated in AI prompts
All third-party document text entering an AI prompt is enclosed in <untrusted_document> tags with a security preamble (ai_json.py:wrap_untrusted()). This is enforced at source-code level and cannot be bypassed by any call-site.

SOP-05

Sanctions & AML screening

Checks vendor entities and associated persons against: OFAC SDN & Consolidated Non-SDN; UN Consolidated List; EU Consolidated Financial Sanctions; OFSI (HM Treasury); and PEP lists. Name normalisation and fuzzy matching applied. Results: Clear, Review (potential match), or Hit (confirmed match).

⚠️
Live feed requirementThe deployment uses representative demo fixtures. For production compliance, the four Tier-1 feeds (OFAC, UN, EU, OFSI) must be connected to direct-issuer downloads. The matching logic, banding and audit trail are production-ready.
A
Algorithm Name normalisation and fuzzy match
Legal suffixes stripped, punctuation collapsed, diacritics transliterated. Fuzzy matching applied per list entry. Match confidence score computed.
S
System Apply outcome and auto-record
⚙ apply_screening_outcome() — registry_service.py:410
  • Clear: all open sanctions IssueRecords for this vendor → status="Closed", closed_reason="re-screened clear"
  • Review: new IssueRecord (kind=sanctions_review) created unless one already open
  • Hit: IssueRecord (kind=sanctions_hit) created; vendor escalated to Critical
  • All bands: VendorScreening record written (timestamp, lists, score, decision)
  • If sanctions.hit notification enabled and result is Hit or Review → notification emitted
  • Audit event written
D
Gate Outcome routing
Hit → Escalate to Compliance; no engagement proceeds
Review → Assessor investigates; issue must be closed before engagement activates
Clear → Open issue auto-closed; vendor eligible for engagement
R-SCR-1No active engagement on open sanctions issue
A vendor with an open sanctions_hit or sanctions_review issue must not have any engagement set to Active. The assessor must close the issue with documented rationale first.

SOP-08

ProAssess 8-stage assessment process

Eight sequential stages; each has a designated lead agent, defined inputs, a defined output, and a system action. All stages run deterministically offline; a live AI model enriches each stage when configured.

📐
Guided by the Methodology folderBoth ProAssess and BRO Chat assess strictly under the firm’s methodology held in the Methodology folder (the Methodology Library, top of the menu, admin-managed). At assessment time the active methodology document(s) are injected into every AI stage as the authoritative directive that overrides general defaults. If no methodology is loaded, both fall back to recognised best practice (PRA SS2/21, EBA Outsourcing, DORA, FSB toolkit, NIST SP 800-161). Maintain the methodology in the Methodology Library — not the general Documents area, which holds per-vendor evidence and does not steer the methodology.

R-IRQ-1Risk-averse gap resolution (enforced algorithmically)
Any domain that cannot be scored is resolved at worst-case value (4). The system does not allow an un-scored domain to deflate the inherent band. Missing data items are logged as GAPs with domain, issue, and required resolution action.
R-IRQ-2Critical control failure floors residual at HIGH
A marginal failure on a critical control floors residual at HIGH regardless of all other control scores. Cannot be overridden without a documented CISO exception on record.

SOP-11

Residual risk & decision

A
Algorithm (Bro — Stage 6) Compute residual band
Residual = inherent band reduced only by evidenced controls. Unverified controls provide zero reduction. Critical control gap floors residual at HIGH regardless of all other controls.
H
Assessor Review, challenge, sign off Stage 6
May override residual band upward only (cannot lower without additional evidence). Override documented with rationale; flagged in audit trail.
A
Algorithm (Bro — Stage 7) Generate decision memo
HIGH=DO NOT PROCEED · ELEVATED=ESCALATE · MODERATE=APPROVE WITH CONDITIONS · LOW=APPROVE. Memo includes: engagement details, risk summary, domain scores, gap register, conditions, rationale.
H
Assessor Sign off and register
Review memo, confirm or override decision with documented rationale, click Register. Triggers full auto-record chain (SOP-12).
LOW — Approve → Active; annual review
MODERATE — Conditions → Active with findings; 6-month review
ELEVATED — Escalate → CISO review; engagement paused
HIGH — Do not proceed → CISO + Legal + CRO sign-off required

24.2 Viny drafting rules (deterministic — code-verified)

R-EXIT-1Strategy type selection
If existing_alternatives list is non-empty → strategy_type = "alternative_provider". If list is empty AND vendor is critical → strategy_type = "multi_source". If non-critical and no alternatives → strategy_type = "alternative_provider" (market scan required and noted in steps).
R-EXIT-2Exit window mandates (CMORG principles)
Critical vendors: planned exit window = 6 months; stressed exit window = 4–6 weeks. Non-critical vendors: planned window = minimum 3 months. These are floors; the plan may specify longer where data portability or service complexity warrants.
R-EXIT-3Missing exit clause — mandatory NOTE (deterministic)
If _contract_has_exit_clause() returns False, Viny must include in the plan rationale: "NOTE: no contractual exit/assistance clause is currently evidenced — confirm termination-for-convenience, transition-assistance and data-return obligations." R-EXIT-4Test cadence and next_test_due auto-computation
Critical vendors: test every 365 days. Standard: 730 days. When a test is logged, next_test_due is automatically computed as last_tested + cadence_days and written to the exit plan record. Late tests reduce the readiness score from 20 to 8 points.
R-EXIT-5Six mandatory transition steps minimum
Viny's deterministic draft always generates exactly 6 transition steps: (1) Invoke exit governance; (2) Confirm/stand up the target arrangement; (3) Extract and validate data; (4) Parallel-run and verify; (5) Cutover and monitor; (6) Decommission and obtain certified destruction. Assessors may add, reorder, or replace steps but may not reduce below 3 without the stressed-playbook readiness points being forfeited.

SOP-27

AI governance

27.1 AI call pipeline — enforced sequence

StepComponentEnforced rule
1Budget checkIf daily budget set, call count checked against cap before the model call. Exceeded: call blocked, last_error set, deterministic fallback runs, ledger row written (success=False).
2SDK checkIf the selected provider's SDK is not installed: call blocked with honest error "SDK not installed". Deterministic fallback runs.
3Provider resolutionPreference order: BRO_LLM_PROVIDER env → first provider with key present. Order: Claude → OpenAI → Grok → Manus → NVIDIA → custom providers.
4System prompt compositionAlways: UNTRUSTED_PREAMBLE + methodology directive + domain prompt from registry. No call-site may bypass this. Prompt source is the editable registry, not hardcoded strings.
5Document injection defenceAll third-party text wrapped in <untrusted_document> tags with security preamble (enforced in ai_json.py:wrap_untrusted()). Cannot be bypassed.
6Output validationparse_json_strict(): strip fences → extract JSON → validate required keys. Failure: one self-repair retry (model corrects its own output). Second failure: deterministic fallback runs, error logged.
7Ledger recordMetadata-only record on every call (success or failure): provider, model, domain, duration_ms, prompt_chars, response_chars, success, redacted error. Never stores prompt text or response content. PII regex-redacted from errors.
R-AI-1Prompts are governed artefacts
The 31-prompt registry is the platform's AI governance artefact. All changes require Administrator access and are audit-logged. The prompt_evals.py golden-set harness must pass before any prompt change ships. Overrides are DB-stored and take effect on the next call — no restart needed.
R-AI-2AI output is advisory only
No AI-generated decision, finding, risk entry, or exit plan takes effect without assessor review and sign-off. AI outputs are labelled in the UI ("AI · web-enriched" badge or "deterministic draft" badge). The assessor's documented judgement is the system of record.

27.2 Dump to Draft — form auto-fill governance

The Dump to Draft function is available on New Vendor, New Engagement, and Exit Plan forms. It accepts uploaded documents and proposes field values derived from the document content. Rules:

  • The system only populates a field when the document clearly supports a value. It never guesses or fabricates.
  • Where options are listed (tier, strategy type), the proposed value must be one of the permitted options.
  • Pre-filled fields are highlighted in amber. The user must review all proposed values before saving.
  • The function is not available on ProAssess or BroAssess forms (those use the full assessment methodology).
  • Offline path (no live model): deterministic label-matching extraction from document text.
  • All Dump to Draft actions are audited under ai.dump_to_draft (fields attempted, fields filled, engine used).

SOP-29

Audit trail & data governance

R-AUD-1Audit log is immutable and comprehensive
Every state-changing action is written to the audit log: actor username, action type, UTC timestamp, and a JSON payload of changed values. The audit log cannot be edited or deleted through any platform interface. Key audit event types covered: all assessment actions, finding and remediation lifecycle, incident actions (including notable flag), all contract events, vendor management including imports, AI key and prompt changes, schedule changes, notification settings changes, authentication events (login, lockout, key saves, provider changes).
R-AUD-2AI call ledger is metadata-only, PII-redacted
The AI call ledger records: provider, model, domain, duration_ms, prompt_chars, response_chars, success, error (redacted). It never stores prompt text or model response content. PII (email addresses, payment card numbers, NINOs, phone numbers) is regex-redacted from error strings before persistence. The ledger is a governance and cost-management tool, not a content log.
R-AUD-3Assessment records are immutable point-in-time snapshots
Every ProAssess registration mints a new, immutable AssessmentRecord (ASM-xxxxxx). Re-running an assessment creates a new record alongside the prior one — it never overwrites it. The full assessment history for any engagement at any point in time is permanently preserved.
R-AUD-4Data persistence on Render
On Render without a persistent disk, the SQLite database resets on every redeploy. A persistent disk must be attached, or the database migrated to Postgres (set env DATABASE_URL), before live operation with real data. The Postgres switch requires only the environment variable — no code changes.
✓
Regulatory evidentiary recordThe audit log, immutable assessment snapshots, AI ledger, document store with full provenance, and finding lifecycle history together constitute the documented third-party risk management record required by DORA Article 28, FCA SYSC 8, and PRA SS2/21. The system is designed to answer the audit question: "what did you know about this vendor, and when?" at any point in time.
